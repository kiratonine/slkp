# ТЗ для фронтенда: поэтапное подключение backend API

Этот файл нужен как рабочий чек-лист для подключения backend во фронтенд. Подключаем всё **поэтапно**, в том же порядке, в котором реализуется backend, чтобы не пропустить важные части и не собирать интерфейс вслепую.

---

## 1. Общая цель

Нужно подключить React frontend к backend API проекта **S1lk x402 Bridge** и постепенно собрать пользовательский flow:

1. регистрация
2. логин
3. получение текущего пользователя
4. получение баланса
5. просмотр и изменение настроек оплат для AI-агентов
6. создание, просмотр и отзыв agent sessions
7. позже — bridge flow, seller flow и agent demo flow
8. просмотр истории bridge payments
9. просмотр детальной информации по конкретному bridge payment
10. просмотр истории ledger
11. просмотр детальной информации по конкретной agent session
12. корректная обработка unified API errors

---

## 2. Базовая подготовка на фронтенде

Что нужно сделать в самом начале:

- завести один общий API client
- завести `.env` для frontend с base URL backend
- сделать единый слой для работы с токеном авторизации
- сделать общий типизированный слой запросов
- сделать общий обработчик ошибок API
- сделать отдельную страницу или секцию для тестирования backend flow
- сделать единый parser для unified API error format
- сделать общий тип ошибки backend:
  - statusCode
  - error
  - message
  - errorCode
  - timestamp
  - path
- предусмотреть отображение бизнес-отказов bridge (`status: rejected`) отдельно от обычных HTTP ошибок

Что важно:

- не хардкодить URL backend
- не дублировать fetch/axios логику по всему проекту
- access token хранить централизованно
- все защищённые запросы отправлять только через общий auth-aware API client

## 3. Порядок подключения endpoint'ов

### Этап A. Аутентификация

Сначала подключить:

- `POST /v1/auth/register`
- `POST /v1/auth/login`

Что нужно сделать во фронтенде:

- страница регистрации
- страница логина
- форма с валидацией полей
- сохранение `accessToken` после логина
- сохранение данных пользователя после логина
- редирект в защищённую часть приложения после успешного входа

Что протестировать:

- регистрация нового пользователя
- ошибка при регистрации существующего email
- логин существующего пользователя
- ошибка при неправильном пароле
- после логина токен реально сохраняется и используется дальше

---

### Этап B. Текущий пользователь

Подключить:

- `GET /v1/users/me`

Что нужно сделать:

- после логина получать актуальные данные пользователя
- сделать guard/проверку авторизованного состояния на фронтенде
- сделать отображение текущего пользователя в интерфейсе

Что протестировать:

- endpoint работает с токеном
- endpoint не работает без токена
- после перезагрузки страницы фронтенд может восстановить состояние пользователя через сохранённый access token

---

### Этап C. Баланс пользователя

Подключить:

- `GET /v1/balance`

Что нужно сделать:

- вывести текущий KZT баланс в личном кабинете
- сделать отдельный виджет или карточку баланса
- предусмотреть состояние загрузки
- предусмотреть пустое/ошибочное состояние

Что протестировать:

- баланс загружается после авторизации
- баланс не запрашивается без токена
- после логина пользователь видит свой стартовый баланс

---

### Этап D. Настройки оплат для AI-агентов

Подключить:

- `GET /v1/settings/agent-payments`
- `PUT /v1/settings/agent-payments`

Что нужно сделать:

- страница или блок “Настройки оплат для AI-агентов”
- переключатель включения/отключения оплаты
- поля дневного лимита
- поля лимита на одну транзакцию
- переключатель `requireConfirmNewSeller`
- кнопка сохранения изменений
- отображение текущих значений из backend

Что протестировать:

- настройки корректно загружаются
- настройки сохраняются
- после обновления страницы приходят обновлённые значения
- frontend не ломается при ошибке backend

### Этап E. Agent Sessions

Подключить:

- `POST /v1/agent-sessions`

Что нужно сделать:

- страница или блок “Agent Sessions”
- список всех созданных session

- отображение:
  - name

  - expiresAt
  - createdAt
  - revokedAt
- кнопка revoke для активной session

- кнопка copy для `sessionToken`
- предупреждение пользователю, что токен нужно сохранить сразу

Что протестировать:

- создание новой session
- отображение токена после создания
- session появляется в списке
- revoke меняет статус session
- повторный просмотр списка показывает актуальный статус

## 4. Как лучше организовать frontend-структуру

Рекомендуемое разделение:

- `services/api` — общий HTTP client
- `services/auth` — register/login/me
- `services/balance` — balance API
- `services/agent-settings` — settings API
- `services/agent-sessions` — sessions API
- `hooks` — React hooks для запросов
- `pages` или `app routes` — страницы интерфейса
- `components` — формы, карточки, таблицы, status badges

Также желательно завести:

- `AuthContext` или аналогичный state layer
- общий `ProtectedRoute` / auth gate
- общий toast/alert слой для ошибок и успехов

## 6. Что важно не забыть

- все запросы после логина должны идти с Bearer token
- состояния loading / error / success должны быть в каждом важном экране
- после logout токен должен очищаться
- sessionToken агента — это отдельный токен, его нельзя путать с accessToken пользователя
- не показывать sessionToken повторно из списка, если backend его не возвращает
- ориентироваться на текущую backend документацию в `docs/api/*.md` и Swagger `/docs`
- bridge payment detail и agent session detail — это отдельные экраны, а не только список
- `bridge/pay` может вернуть не только HTTP error, но и business response со `status: rejected`
- `solanaTxSignature` нужно показывать в UI отдельно, с возможностью copy
- sessionToken показывается только в момент создания session и должен иметь заметную кнопку copy
- unified API errors должны отображаться одинаково во всём приложении

## Единая обработка ошибок

В бэкэнде теперь используется единый формат обработки ошибок API для HTTP-ошибок.

Фронтенд должен поддерживать следующую структуру:

```json
{
"statusCode": number,
"error": "string",
"message": "string",
"errorCode": "string (optional)",
"timestamp": "ISO string",
"path": "string"
}
```

Важно:

- не полагайтесь только на `message`
- если `errorCode` существует, используйте его в логике пользовательского интерфейса, когда это уместно
- `POST /v1/bridge/pay` также может возвращать бизнес-ответ:
  - `status: "ok"`
  - `status: "rejected"`

Этот бизнес-ответ должен обрабатываться отдельно от обычных HTTP-ошибок.
